## Ways to Contribute

You can help the project by:

- Improving documentation (`README.md`, `docs/`, examples)
- Reporting bugs or unexpected behavior
- Suggesting new language features or syntax
- Improving tooling (CLI, formatter, editor support)
- Writing tests or example programs
- Reviewing pull requests and discussions

---

## Getting Started

1. **Fork** the repository on GitHub
2. **Clone** your fork locally:
   ```bash
   git clone https://github.com/your-username/starlight-programming-language.git
   cd starlight-programming-language


3. Create a new branch for your change:

   ```bash
   git checkout -b feature/my-change
   ```

## Documentation Contributions

Documentation is extremely important for Starlight.

Guidelines:

* Use clear, simple language
* Include code examples where helpful
* Prefer `.md` files in the `docs/` directory
* Keep formatting consistent (Markdown)

If adding a new doc:

* Use descriptive filenames
* Link it from `README.md` or `docs/README.md`

---

## Feature Requests & Proposals

For new language features:

1. Open a **GitHub Issue**
2. Describe:

   * The problem you are solving
   * Proposed syntax (if applicable)
   * Example code
   * Expected behavior
3. Keep proposals focused and incremental

Large changes may require discussion before implementation.

---

## Bug Reports

When reporting bugs, please include:

* Starlight version (if applicable)
* Operating system
* Minimal reproducible example
* Expected vs actual behavior
* Error messages or stack traces

---

## Pull Request Guidelines

Before submitting a pull request:

* Ensure your changes are well-scoped
* Follow existing project conventions
* Update documentation if behavior changes
* Keep commits clean and descriptive

Pull Request titles should be clear:

```
Add range() documentation
Fix import behavior in examples
Improve error message wording
```

---

## Code Style

If contributing code:

* Prefer clarity over cleverness
* Avoid breaking changes without discussion
* Keep changes readable and well-commented
* Match existing naming conventions
